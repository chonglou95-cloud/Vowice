# Vowice 调研与决策记录

## 已确认信息

- 产品名：声契 Vowice。
- 品牌含义：Vow + Voice，由声音承载的承诺。
- 品牌标语：听见他，也被他记住。Hear him. Be remembered.
- 产品定位：一段存在于声音里的记忆。A relationship made of voice.
- 用户方向：以女性用户为主的 AI 虚拟恋人与情感陪伴。
- 交互特点：纯语音为核心，强调声音定制、长期记忆和关系沉淀。
- 团队：1 名产品经理 + 1 名开发。
- 时间：7 天，每人每天 4–5 小时。
- 交付：可运行 MVP、可交互原型、PRD、前期用户调研。
- 用户明确补充：不接受“用户发一条语音、AI 回一条语音”作为产品全部，优先考虑成品效果和玩法完整度。
- 团队将使用 Codex 与 Claude Agent 长时间辅助开发，人类成员的 4–5 小时主要用于决策、调研、验收和集成。

## 待调研

- 中文 ASR、LLM、TTS 和长期记忆的低成本集成方案。
- 开源语音模型的商用许可与伦理边界。
- 最适合一周 MVP 的客户端和后端技术栈。
- 应进入 MVP 的核心玩法与明确非目标。

## 技术调研初步结论

- 客户端建议使用 Expo + React Native + TypeScript。`expo-audio` 官方库支持录音与播放，适合快速完成 iOS/Android 可运行原型。
- 后端建议使用 FastAPI，便于串联 Python 语音模型、LLM API 与记忆服务。
- 数据层建议使用 Supabase Postgres。其 pgvector 扩展可存储 embedding 并做向量相似度检索，可用于长期记忆召回。
- ASR 首选 `faster-whisper`：MIT 许可，支持 CPU/GPU 量化，官方资料显示其相比原 Whisper 实现更快且更省内存。MVP 先使用 `small` 或 `medium` 模型。
- 对话模型首选 Qwen3 的非思考 Instruct 模型。官方仓库表明开放权重为 Apache 2.0，并强调多轮对话和角色扮演能力。MVP 建议调用托管 API，不在本地部署大参数模型。
- TTS 首选验证 CosyVoice 2/3。官方仓库支持中文、零样本声音、指令控制和服务部署，代码许可为 Apache 2.0。但本地运行依赖较多，应在第 1 天设置 Go/No-Go 技术检查。
- 交互架构建议先做半双工：用户按住说话，结束后经 ASR → LLM → TTS 返回语音。一周内不做实时全双工、打断、回声消除与端侧模型。
- 上述半双工仅是技术链路，不代表产品只有单轮语音问答。产品必须形成“主动陪伴 → 连续通话 → 记忆沉淀 → 下次主动陪伴”的闭环。

## 产品玩法方向

- **声契工坊**：通过预设音色、文字描述和特质控件创建一个有辨识度的虚拟恋人声音，支持即时试听和保存版本。
- **关系共识**：设置称呼、性格、沟通方式、主动联系频率、禁忌话题和边界，形成一份可修改的“声契”。
- **连续语音通话**：以“通话”而非“发消息”的心智模型呈现，支持多轮上下文、会话状态和结束仪式。
- **情境声场**：提供深夜、通勤、雨声、睡前等声场与对话节奏，用声音营造场景，增强纯语音产品的沉浸感。
- **共同记忆**：每次通话后提取 1–3 条候选记忆，用户可确认、编辑、忽略或删除；经确认的记忆参与后续对话。
- **回声日记**：AI 以第一人称留下一段短语音和一句关系摘要，用户可回听某个时刻。
- **主动声音胶囊**：基于用户允许的时段与已确认记忆，AI 主动留下早安、晚安或事件跟进语音。
- **关系时刻**：以时间线展示第一次通话、重要约定、已完成事件与纪念日，不使用亲密度分数制造情感压力。

## 待验证风险

- CosyVoice 预训练权重、附加资源与实际托管平台的许可需逐项留档，不能仅根据代码仓库许可做商用结论。
- 若 CosyVoice 无法在 2 小时内跑通，MVP 需立即切换至带合法预设音色的托管 TTS，并把完整自定义声音放入下一迭代。

## 最终方案决策

- 本周主链路使用阿里云百炼托管 ASR、Qwen 对话与 CosyVoice TTS，以获得较快且稳定的中文效果。
- 后端必须封装 ASR/Chat/TTS Provider 接口，后续可替换为 faster-whisper、Qwen3 开放权重与 CosyVoice3 自托管。
- 百炼官方文档显示 CosyVoice v3.5 系列支持声音设计、声音复刻与指令控制，可直接把用户的语义化调音控件映射到 TTS 指令。
- 开源备选中，faster-whisper 为 MIT，Qwen3 开放权重为 Apache 2.0，CosyVoice 代码与官方 CosyVoice3 权重页面标注 Apache 2.0；ChatTTS 模型为 CC BY-NC 4.0，不纳入未来商业发布主链路。
- Apple 审核指南要求对与第三方 AI 共享个人数据进行明示并取得许可，同时要求隐私政策、数据删除与最小化。
- 《人工智能生成合成内容标识办法》已于 2025-09-01 施行，产品需在架构中预留音频显式与隐式标识。

## 问卷设计决策

- 问卷目标用时为 5–8 分钟，采用 4 步、13 题结构，其中 11 题必答、2 题选填。
- 研究重点是使用场景、声音调节偏好、玩法优先级、主动联系边界、长期记忆控制和主要顾虑。
- 按用户最新要求，问卷为一次性匿名研究，不包含后续访谈邀请，也不收集任何联系方式。
- 数据收集支持本地演示和 Supabase 正式模式；正式模式使用 RLS 和最小权限，仅允许匿名插入，不允许网页读取历史答案。
- 首页年龄确认与研究同意合并为一次主动勾选，减少操作但不预先替受访者同意。
- 将含义模糊的“温度感”改为“声音气质（温柔或清冷）”，将“声音工坊”候选名改为更贴近关系叙事的“初声”。

## 代码仓库决策

- Vowice 使用单一私有主仓库统一管理用户研究、产品文档、原型、MVP 代码和作品集材料。
- 远程仓库：`https://github.com/chonglou95-cloud/vowice`，默认分支为 `main`。
- GitHub 只保存项目代码与文档，不保存受访者答案、原始研究数据或真实密钥。
- 分享问卷仍需单独配置 HTTPS 托管和 Supabase 数据库，私有主仓库不直接承担问卷数据存储。
