# Vowice PM 作品集素材与叙事计划

## 1. 这个项目应证明什么

Vowice 案例要证明你作为产品经理能够：

1. 在大厂已有通用 AI 能力的市场中，找到可被小团队做深的差异化场景。
2. 把模糊的“情感陪伴”拆成声音身份、主动陪伴、关系记忆和边界修复。
3. 理解 ASR、LLM、TTS、记忆、延迟、成本和许可证之间的产品取舍。
4. 用真实用户证据修改方案，并说明竞品研究如何影响决策。
5. 在敏感品类中同时考虑用户价值、关系信任、数据和安全边界。

## 2. 推荐的案例结构

### 01 项目概览

- 产品：声契 Vowice。
- 定位：面向成年女性的声音优先 AI 长期恋人。
- 人员：产品经理 1 人、开发 1 人，Codex/Claude 辅助。
- 周期：标注实际日期和投入。
- 你的贡献：研究、定位、优先级、交互、PRD、验收与迭代。

### 02 理解问题

- 为什么“通用 AI 可以聊天”不等于“一段有连续性的关系”。
- Gooda 竞品分析中的关键证据：活人感有价值，但声音、记忆、OOC 和付费体验仍有明显缺口。
- 用户访谈中出现的行为场景、当前替代方案与关键原话。

### 03 探索方案

展示至少三个方向的比较：

| 方向 | 价值 | 问题 | 决策 |
|---|---|---|---|
| 做更强的角色聊天 | 使用门槛低 | 容易被豆包等大厂覆盖 | 不作为主差异 |
| 复制 Gooda 的朋友圈与日记 | 关系存在感强 | 同质化、功能庞大、偏离纯语音 | 只学关系连续性机制 |
| 声音身份 + 关系记忆 + 主动仪式 | 与团队能力和品牌一致 | 对 TTS 稳定、延迟和数据权限要求高 | **选择** |

### 04 决策与取舍

- 为什么本周用托管 API，却仍要设计可替换 Provider 层。
- 为什么先做用户可确认的结构化记忆，而不是不透明的无限自动记忆。
- 为什么暂不允许复制任意真人声音。
- 为什么主动陪伴由用户设定频率，不用情感责怪换留存。

### 05 产品设计

- 信息架构与核心流程。
- 声契工坊、通话、记忆和睡前来信的关键界面。
- 后端 ASR→LLM→TTS 与记忆的架构图。
- 可删除记忆、修复错误和主动频率等信任设计。

### 06 反馈与迭代

- 可用性测试任务、成功率和关键问题。
- 展示至少一个由用户证据触发的明确改动。
- 展示迭代前后对比和修改依据。

### 07 结果与反思

- 本周实际完成了什么，哪些仍是原型或假数据。
- 实测的延迟、任务完成率、满意度和问题数量。
- 如果有 4 周，下一步最需要验证什么。
- 哪个原始假设被证伪或大幅修正。

## 3. 从第一天开始保留的素材

| 素材 | 说明 |
|---|---|
| 日期化决策日志 | 决定了什么、证据是什么、放弃了什么 |
| 调研原始记录 | 匿名编号、原话、行为和反证 |
| 优先级草稿 | 展示功能如何从广到窄 |
| 原型迭代 | 保留 v0.1、v0.5、v1.0 关键截图 |
| 工程取舍 | 模型对比、延迟、成本、许可证和失败记录 |
| QA 证据 | 真机、弱网、权限被拒、模型超时和删除数据 |
| Demo 素材 | 干净的屏幕录制、真实生成音频和字幕 |
| 贡献边界 | 明确 PM、开发、Codex 和 Claude 分别做了什么 |

## 4. 面试高频追问

1. 你们怎么证明女性用户真的需要这个产品？
2. 这和豆包自定义智能体的根本差别是什么？
3. 为什么你们选声音而不是视觉角色？
4. 用户愿意调节这么多声音参数吗？
5. 长期记忆出错怎么办？为什么不全部自动保存？
6. 主动陪伴和情感操纵的分界是什么？
7. 你们的模型不是自己训练的，护城河在哪里？
8. 为什么本周不做全双工实时通话？
9. 如何衡量“被记住”这个抽象感受？
10. 这个项目中哪个决策是你亲自做的，不是 Agent 做的？

## 5. 案例表述原则

- 不声称一周 MVP 已经证明商业成功。
- 不把 AI 生成的文档数量当作 PM 成果。
- 不伪造用户量、留存、付费意愿或访谈引用。
- 把限制、失败和改变想法的瞬间放进案例；它们比“从头到尾都很顺利”更能证明产品判断。
