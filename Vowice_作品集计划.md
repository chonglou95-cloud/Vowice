## 1. 7 天计划假设

- 执行周期：Day 1–Day 7；若从 2026-06-19 开始，则对应 6 月 19 日至 6 月 25 日。
- 人员：1 名产品经理 + 1 名开发者。
- 人工投入：每人每天 4–5 小时。
- Agent 投入：Codex 与 Claude 可执行长时间代码、资料和评审任务，但必须由人员做需求决策、真机验收和用户研究。
- 最终交付：可运行 MVP、可交互原型、PRD v1.0、真实用户调研与作品集素材。

## 2. 本周成功标准

### 产品成功标准

1. 新用户能在 5 分钟内完成声音与关系创建。
2. 用户能完成一次至少 3 轮的连续语音通话。
3. 通话结束后生成 1–3 条候选记忆，用户可确认、修改、忽略和删除。
4. 已确认记忆能在下一次对话或主动声音中被正确使用。
5. 产品至少呈现一种“聊天之外的关系”：睡前来信、主动声音胶囊或回声日记。

### 工程成功标准

1. 主要流程在真机或 iOS 模拟器连续完成 5 次，无 P0 崩溃。
2. 10 秒用户语音在普通网络下的完整回复目标为 8 秒内开始播放；若未达标，必须记录 ASR、LLM 和 TTS 分段延迟。
3. 语音录音、模型错误、超时和网络断开有可恢复状态。
4. 原始用户录音默认不长期保存；测试数据可一键清除。

### PM 作品集成功标准

1. 至少 5 名目标用户的真实访谈或可用性测试记录。
2. 保留“原始假设 → 用户证据 → 产品决策 → 上线结果 → 复盘”完整证据链。
3. PRD 中每个 P0 需求都有验收标准，并与原型页面和开发 Issue 对应。
4. 有 60–90 秒主流程 Demo 和 5–8 张可用于作品集的关键截图。

## 4. 夜间 Agent 节奏

1. 每天结束前，产品经理和开发者只提交一份已排序任务列表。
2. 每个任务必须包含背景、输入、预期结果、验收标准和不能修改的范围。
3. Codex 负责主分支上的实现；Claude 默认只做备选方案、Prompt 和审查，不同时修改同一批文件。
4. 次日开工先读测试、diff 和进度文件，人工验收后才合并。
5. 禁止 Agent 自行宣布用户研究结论；它可以编码、归类和挑战结论，但证据和决策归 PM。

## 5. 风险与止损点

| 风险 | 早期信号 | 止损方案 |
|---|---|---|
| 声音模型无法准时跑通 | Day 1 两小时后仍无法稳定合成中文 | 立即切换至阿里云百炼 CosyVoice 托管 API；自托管放入后续路线图 |
| 端到端延迟过高 | 第一段音频超过 10 秒才播放 | 使用流式 TTS、缩短回复、显示可感知的倾听/思考状态 |
| 声音定制不稳定 | 同一配方多次生成音色差异明显 | 固定 voice ID 与声音配方，把动态参数仅用于语速、情绪和表达风格 |
| 功能多但无闭环 | 每个页面都只能单独演示 | 优先保证“创建→通话→记忆→主动陪伴”端到端可运行 |
| Agent 并行造成代码冲突 | 两个 Agent 修改同一文件或架构 | 一项任务只有一个实现者，另一个仅评审；每日设置合并时间 |
| 用户调研变成为方案背书 | 只问“你喜欢吗”或样本全是朋友 | 使用行为回溯问题，同时招募 AI 陪伴用户和非用户，保留反例 |
